C++ 内存对齐详解

一、什么是内存对齐

内存对齐（Memory Alignment）是指数据在内存中的地址必须是某个特定值的倍数。这个特定值称为对齐边界（Alignment Boundary）。

1.2 输出的地址是虚拟地址

程序输出的地址（如 0x7ffeefbff56c）是虚拟地址，不是物理内存的实际地址。

虚拟地址 vs 物理地址

| 概念 | 说明 |

|——|——|

| 虚拟地址 | 程序看到的地址，每个进程有独立的虚拟地址空间 |

| 物理地址 | 内存条上的实际地址 |

| MMU | CPU 中的内存管理单元，负责虚拟地址到物理地址的转换 |

地址转换过程

为什么使用虚拟地址？

| 优势 | 说明 |

|——|——|

| 内存保护 | 每个进程的地址空间隔离，防止越界访问 |

| 内存复用 | 多个进程可以使用相同的虚拟地址，映射到不同物理地址 |

| 内存管理 | 支持分页、分段、内存映射等高级功能 |

| 程序移植性 | 程序无需关心物理内存布局 |

注意：在用户态程序中，无法直接获取物理地址，这是操作系统的安全机制。

二、为什么需要内存对齐

2.1 硬件层面的原因

  1. 性能提升：CPU 读取内存是以字（Word）为单位的，通常是 4 或 8 字节。如果数据未对齐，CPU 可能需要多次读取。

  2. 硬件要求：某些硬件架构（如 ARM、SPARC）要求特定类型的数据必须对齐，否则会触发硬件异常。

4.4 数组的对齐规则

核心原则：数组的对齐要求 = 其元素类型的对齐要求。

基本示例

| 属性 | 说明 |

|——|——|

| alignof(arr) | 等于 alignof(元素类型) |

| sizeof(arr) | 等于 元素数量 × sizeof(元素类型) |

| 地址要求 | 保证第一个元素对齐，后续元素自动对齐 |

数组元素的自动对齐

强制对齐数组

| 属性 | 值 | 说明 |

|——|—–|——|

| alignof(arr) | 4 | 仍然是元素类型的对齐 |

| 实际地址 | 必须是 16 的倍数 | alignas 强制要求 |

| sizeof(arr) | 40 | 大小不变 |

关键理解：alignas 只改变首地址

alignas 对于数组来说，只会改变首地址的对齐要求，不会改变数组内部的任何东西。

内存布局示意
对比普通数组

| 特性 | 普通数组 int arr[10] | 对齐数组 alignas(16) int arr[10] |

|——|———————–|————————————|

| 首地址 | 4 的倍数 | 16 的倍数 |

| 元素大小 | 4 字节 | 4 字节 |

| 元素间距 | 4 字节 | 4 字节 |

| 数组大小 | 40 字节 | 40 字节 |

| alignof(arr) | 4 | 4 |

唯一区别

一句话总结：alignas 对于数组，只保证首地址对齐，不改变数组的内部结构。

为什么需要这个？

通常用于需要特定对齐的场景：

  • SIMD 指令集（如 ARM NEON、x86 SSE）

  • DMA 操作（直接内存访问）

  • 缓存行优化（提高缓存命中率）

数组 vs 结构体：alignas 的区别

alignas 对数组和结构体的影响完全不同，这是一个容易混淆的点。

对比表格

| 属性 | alignas(16) int arr[10] | struct alignas(16) S { char data[10]; }; |

|——|—————————|——————————————-|

| alignof | 4（元素类型的对齐） | 16（结构体的对齐） |

| sizeof | 40（数据大小） | 16（含填充） |

| 首地址 | 16 的倍数 | 16 的倍数 |

| 内部结构 | 不变 | 可能添加填充 |

关键区别

| 概念 | 数组 | 结构体 |

|——|——|——–|

| alignas 的作用 | 只影响变量的首地址 | 同时影响类型的对齐和大小 |

| alignof | 返回元素类型的对齐 | 返回指定的对齐值 |

| sizeof | 返回数据实际大小 | 返回向上取整后的大小（含填充） |

| 填充字节 | 无 | 可能添加填充 |

为什么会有区别？

  • 数组：alignas 只是对变量的约束，不改变数组类型本身的属性

  • 结构体：alignas 是对类型的约束，改变类型的对齐要求，进而影响大小

一句话总结：alignas 对数组是”外在约束”，对结构体是”内在属性”。

5.3 alignas 对结构体大小的影响

为什么 sizeof 和 alignof 都是 16？

原因分析

alignas(16) 同时影响了结构体的对齐要求和大小：

| 影响 | 说明 |

|——|——|

| 对齐要求 | 结构体起始地址必须是 16 的倍数 |

| 大小要求 | 结构体总大小必须是 16 的倍数（否则数组元素无法对齐） |

内存布局

对比没有 alignas 的情况

| 属性 | AlignedStruct | NormalStruct |

|——|—————–|—————-|

| alignof | 16 | 1 |

| sizeof | 16 | 10 |

关键规则

  1. 对齐要求：alignof = max(N, 成员最大对齐)

  2. 大小要求：sizeof = ceil(成员总大小 / N) × N

5.4 char 类型数组与强制对齐

char 类型的对齐要求是 1，但当它作为 alignas(16) 结构体的成员时，结构体的强制对齐会带动成员的实际地址也满足对齐。

| 情况 | char 数组的对齐要求 | 实际地址 |

|——|———————-|———|

| 单独声明 char arr[10] | 1（任意地址） | 不确定 |

| 在 alignas(16) 结构体中 | 1（任意地址） | 一定是 16 的倍数 |

8.3 为什么继承是安全的，而 reinterpret_cast 是危险的？

核心区别

| 方式 | 类型 | 编译器保证 |

|——|——|———–|

| reinterpret_cast<Base*>(&d) | 未定义行为 | ❌ 编译器不保证任何事 |

| Base* b = &d（继承） | 标准行为 | ✅ 编译器必须保证正确 |

继承的内存布局是标准定义的

C++ 标准规定：派生类对象中包含一个完整的基类子对象。

向上转型是安全的

| 步骤 | 说明 |

|——|——|

| &d | 获取 Derived 对象地址 |

| 隐式转换 | 编译器知道 Base 子对象在 Derived 中的位置 |

| 结果 | b 指向 Derived 对象中的 Base 子对象 |

编译器保证：b->b 访问的一定是 Base::b，不会出错。

reinterpret_cast 的问题

| 问题 | 说明 |

|——|——|

| 无继承关系 | 编译器不知道 Base 和 Derived 的关系 |

| 纯粹的内存重解释 | 只是”强行”把内存当作 Base 类型 |

| 依赖巧合 | 两个结构体布局相同才能”工作” |

| 修改即崩溃 | 修改任一结构体，布局可能不同 |

对比总结

| 特性 | 继承 | reinterpret_cast |

|——|——|——————-|

| 类型安全 | ✅ 是 | ❌ 否 |

| 编译器保证 | ✅ 标准 | ❌ 未定义 |

| 修改结构体 | ✅ 仍然正确 | ❌ 可能崩溃 |

| 可维护性 | ✅ 好 | ❌ 差 |

九、总结

| 概念 | 说明 |

|——|——|

| 内存对齐 | 数据地址是对齐边界的倍数 |

| 虚拟地址 | 程序输出的是虚拟地址，由 MMU 转换为物理地址 |

| sizeof | 查询类型占用的总字节数 |

| alignof | 查询类型的对齐要求 |

| alignas | 指定变量或类型的对齐要求 |

| 结构体对齐 | 成员按自身对齐，整体按最大对齐对齐 |

| 数组对齐 | 数组的对齐要求 = 元素类型的对齐要求 |

| 未对齐访问 | 可能导致性能下降或硬件异常 |

| __attribute__((packed)) | GCC/Clang 取消结构体对齐 |

| aligned_alloc | C11 标准的对齐内存分配函数 |

| 继承 vs reinterpret_cast | 继承是安全的，reinterpret_cast 依赖内存布局巧合是危险的 |
